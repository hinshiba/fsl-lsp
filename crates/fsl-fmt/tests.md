# fsl-fmt テストリスト

t_wada 流 TDD のテストリスト。1 サイクル (Red → Green → Refactor) ごとに 1 項目を消化し、チェックボックスを ✅ に更新する。新しい必要が判明したら本リストに追記する。

## 公開 API

`pub fn format(src: &str) -> Option<String>`

- 成功時: `Some(整形済みソース)`（末尾は改行 1 つで終わる）
- 失敗時: `None`（パースエラー／レキサーエラーがあれば部分整形しない）

## 正常系

### 最小ケース
- [x] 空ソース `""` → `Some("")`
- [x] 空モジュール `module M {}` を整形して返す
- [x] 空 trait `trait T {}` を整形して返す

### インデント
- [x] モジュール本体は 2 スペースでインデント
- [x] always / initial ブロック内は基底 + 2 スペース
- [x] ネストブロック (`if` / `par` / `seq` / `match`) は深さに応じて 2 スペース × n

### 空白
- [x] 二項演算子 (`+`, `-`, `==`, `&&`, `||`, ほか) の前後にスペース 1 つ
- [x] 単項演算子 (`!`, `-`) と被演算子の間にスペースなし
- [x] カンマの後にスペース 1 つ、前にスペースなし（関数呼び出し / タプルで検証）
- [x] 型注釈の `:` は前にスペースなし、後にスペース 1 つ
- [x] `:=`（reg 代入）の前後にスペース 1 つ

### ブロック・ブレース
- [x] K&R: 開きブレース `{` は前トークンと同じ行
- [x] 閉じブレース `}` は独立行、対応する開きと同じインデント

### 区切り
- [-] セミコロンは省略可能位置でも挿入する（plan.md 準拠）— **MVP 延期**: 一般ブロックでは最終式が値となる可能性があり、無条件挿入は意味を変える恐れがある。
- [x] CompilationUnit のトップレベルアイテム間は空行 1 つ（**新規空行ポリシーで上書き、下記参照**）

### 改行
- [x] 出力末尾は改行 1 つで終わる（全テストで検証）
- [x] 連続する空行は最大 1 つに圧縮（パーサが改行を破棄するため出力構造的に自明）— **新規空行ポリシーで上書き、下記参照**

### 各構文要素
- [x] `reg name: Bit(8)` 宣言を整形
- [x] `input` / `output` / `mem` 宣言を整形
- [x] `def f(a, b): T = expr` を整形
- [x] `def f(...): T seq { ... }` を整形
- [x] `def f(...): T par { ... }` を整形
- [x] `match` 式: `case pat => body` 各アームを整形
- [x] `if (cond) then-expr` を整形（else なし）
- [x] `if (cond) then-expr else else-expr` を整形
- [x] `val name = expr` 宣言を整形
- [x] タプル型 `(T1, T2)` を整形
- [x] `Array[T]` / `List[T]` / `Bit(N)` 型を整形

### 演算子優先順位
- [x] 必要な括弧は維持、不要な括弧は除去しない（パース木の構造を保つ）
- [x] **`~(a ^ b)` 等、unary の binary 子は必ず括弧で囲む**: unary は binary より結合度が強いため、括弧を外すと AST が変わる
- [x] **`~(|x)` 等、unary の unary 子は必ず括弧で囲む**: FSL コンパイラは `~|x` のような unary 連続をサポートしないため

### 行幅
- [x] 100 文字以内に収まる行はそのまま 1 行（既定動作）
- [-] 100 文字超の関数呼び出しは引数で改行（ソフトラップ）— **MVP 延期**: 計測＋分割の実装が必要
- [-] 関数呼び出しのソフトラップは `func(` 直後・`)` 直前で改行しない（FSL パーサ制約） — 上記延期に紐付く
- [-] 引数が 1 行に収まらない場合、第 1 引数は `func(` と同じ行に置き、第 2 引数以降をカンマ直後で改行 — 上記延期に紐付く

### コメント保存
- [x] 単独行の `//` コメントを直前／直後の構文要素の直上行として保持
- [x] 単独行の `/* ... */` コメントを保持
- [x] 式中コメントは整形上の都合で式の前後または直上行に移動してよい（消失させない）
- [x] **式インラインコメントは直上行に移動する（下ではない）**: `val z = a /* tag */ + b` → `/* tag */` を `val z = a + b` の上に
- [x] **ブロック境界 `{ ... /* foo */ }` を越えてコメントが移動しない**: ブロック内末尾のコメントは閉じ `}` の直前に残す
- [x] 行末コメント `} // add` のように直前の `}` が閉じるブロックの「親」スコープにとどまる（次の閉じ `}` までに排出）
- [x] 空ブロック内にコメントがある場合は複数行 `{ コメント }` 形式に展開する
- [x] ~~**物理行末トレイリングコメント `input a: Bit(8) // hoge` を直上行に移動**: AST span の外でも同じ物理行にあるコメントは文に属するとみなす~~ — **新規仕様で同一行維持に変更、下記参照**

## 異常系

- [x] パースエラーあり → `None`
- [x] レキサーエラーあり → `None`
- [x] 不完全モジュール `module M {` → `None`
- [x] 不正トークン (例: `@@@`) → `None`
- [x] 空ソースは異常ではない（`Some("")` を返す）

## 空行ポリシー（新規）

ソースの空行を最大 1 行まで保存。並び替えグループ境界では 2 行強制。

- [x] 連続空行 0 行（隣接して書かれた 2 文）→ 出力でも 0 行（同一グループ内）
- [x] 連続空行 1 行 → 出力でも 1 行
- [x] 連続空行 2 行（同一グループ内）→ 1 行に圧縮
- [x] 連続空行 3 行以上（同一グループ内）→ 1 行に圧縮
- [x] コメントは空行カウントの区切りになる: コメント前の空行とコメント後の空行は独立にカウント（count_blank_lines_between でコメントを除外してセグメント加算）
- [x] トップレベル item 間 (`module A {} module B {}` のような同一行記述) → 0 行（ソースに空行なし）
- [x] トップレベル item 間の連続空行も同様にクランプ
- [x] グループ境界（ソース順維持時）→ ソース空行を max 2 で保存（1 行ソース → 1 行出力、2+ 行 → 2 行）
- [x] グループ境界（並び替え発生時）→ 強制 2 空行（視覚分離のため）
- [x] 単一グループしか存在しないモジュール → グループ境界の空行は出ない

## インライン末尾コメント維持（新規）

物理行末（同一行末尾）のコメントはインライン維持し、`//` 開始列 (0-indexed) を偶数に揃える。式中コメントの挙動は変更しない。

- [x] `val x = e // msg` を整形 → 同一行で `// msg` 維持
- [x] `val x = e /* msg */` → 同一行で `/* msg */` 維持
- [x] `reg name: Bit(8) // msg` → 同一行維持
- [x] `mem [Bit(8)] m(16) // msg` → 同一行維持
- [x] `input a: Bit(8) // hoge` → 同一行維持（既存挙動から差し替え）
- [ ] **アラインメント**: `//` 開始列 (0-indexed) は偶数。最低 1 スペース、必要なら +1 スペース
  - 例: `val x = 1 // tag` （expr 末 col 9、`//` col 10 偶数、1 スペース）
  - 例: 偶数長の expr が末尾 → 2 スペース必要
- [ ] インデントありの宣言でも同じ規則（インデント込み絶対列で偶数）
- [x] 式中コメント `val z = a /* tag */ + b` は **従来通り直上に移動**（変更なし）
- [x] ブロック境界のコメント挙動は **変更なし**

## モジュールアイテム並び替え（新規）

モジュール本体のフィールドを 4 グループに安定ソート。各グループ間に空行 2 行。

- グループ定義:
  - G1: `Input`, `Output`
  - G2: `OutputFn`（出力関数）
  - G3: `Reg`, `Mem`, `Val`, `NewInstance`, `Composite`
  - G4: `Fn`, `Stage`, `Always`, `Initial`

- [x] G1→G2→G3→G4 の順に並ぶ
- [x] 同一グループ内は原順序を保つ（安定ソート）
- [x] グループ間は空行 2 行
- [x] G1 のみのモジュール → グループ境界の空行は出ない
- [x] G4 のみのモジュール → 同上
- [x] `input` と `output` は同じ G1 なので並び替わらない
- [x] `def` と `stage` と `always` と `initial` は同じ G4 なので並び替わらない
- [x] ソース順が逆転している場合（例: `def` の下に `input`）→ I/O が上に並ぶ
- [x] アイテムの直上スタンドアロンコメントは対象アイテムに追従して移動
- [x] アイテムのトレイリングコメントも対象アイテムに追従して移動
- [x] `fsl-sample/add32.fsl` 並び替え後も再パース可能
- [x] `fsl-sample/test_alu32.fsl` 並び替え後も再パース可能

## 未実装 Field の最低限書き出し（新規 / 既存 TODO 解消）

並び替え対象になるため、現状 `_ => {}` 扱いの Field 種を書けるようにする。

- [x] `output def name(params): T` → 出力（`OutputFn`、戻り型あり）
- [x] `output def name()` / `output def name(): Unit` → 戻り型省略 / `Unit`
- [x] `stage name(params) { body }` → 出力（`Stage`）
- [x] `type name(field: T, ...)` → 出力（`Composite`）
